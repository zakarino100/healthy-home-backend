# Healthy Home OS — Inbound AI Phone Receptionist Build Spec
**Version:** 1.1 (patched per Robin review)
**Target:** Replit Agent (Healthy Home Backend project)  
**Depends on:** Healthy Home OS backend (base spec) must be complete and running  
**Architecture approved by:** Robin

---

## READ THIS FIRST — Instructions for Replit Agent

You are adding an inbound AI phone receptionist to the existing Healthy Home backend. This feature must be as self-contained as possible. Only touch existing files where explicitly required by this spec.

The receptionist will:
1. Answer inbound calls with an ElevenLabs voice
2. Greet the caller and capture the reason for their call
3. Warm-transfer the call to the owner with a whispered summary before connecting
4. Log every call to the database

**Build in the order listed. Do not skip phases.**

---

## Safety / Data Protection Rules

- Do not delete, drop, or recreate any existing tables, models, or routes
- Do not modify any existing backend files except `server.js` (one new import line only) and `prisma/schema.prisma` (one new model only)
- Frontend changes are limited to adding the new call log page and the minimum routing/navigation needed to access it
- Do not break any existing leads, jobs, customers, canvassing, or scheduling behavior
- All schema changes must be additive — one new model only, no field changes to existing models
- This feature must be logically isolated so it can be disabled by removing/unregistering the phone routes and related UI without affecting any other Healthy Home functionality. Database artifacts such as the additive `CallLog` table may remain.
- Do not overbuild — this is a receptionist, not a booking agent or CRM assistant

## Build Phase 0 — Read Vapi Docs Before Writing Any Code

**This phase is mandatory. Do not skip it.**

Before writing any webhook handlers or assistant config responses, fetch and read the current Vapi documentation at `https://docs.vapi.ai` to confirm:

1. The exact webhook event name Vapi sends when an inbound call needs an assistant (may be `assistant-request` or similar — confirm actual name)
2. The exact webhook event name(s) Vapi sends when a call reaches its final state (may be `call.ended`, `end-of-call-report`, or similar — confirm actual name and which event carries transcript + summary)
3. The exact shape of the final call webhook payload — specifically confirm field paths for:
   - caller phone number
   - call start time
   - call end time

   - Vapi's internal call ID
4. The exact format of the assistant configuration response that Vapi expects from an `assistant-request` webhook

## Build Phase 1 — Schema Change

**File:** `prisma/schema.prisma`

Add this single new model. Do not modify any existing models.

```prisma
model CallLog {
  id               String    @id @default(uuid())
  providerCallId   String?   @unique  // Vapi's internal call ID — used as idempotency key
  callerPhone      String?
  startedAt        DateTime?
  endedAt          DateTime?
  durationSeconds  Int?
  transcript       String?   // full conversation transcript from Vapi
  summary          String?   // AI-generated summary of reason for call
  category         String?   // optional — may remain null in MVP
  transferStatus   String?   // "transferred" | "no_answer" | "voicemail" | "not_attempted"
  answeredByOwner  Boolean   @default(false)
  rawPayload       Json?     // full Vapi webhook payload — stored for debugging
  createdAt        DateTime  @default(now())
  updatedAt        DateTime  @updatedAt
}
```

After adding the model:
1. Run `npx prisma migrate dev --name add_call_log`
2. Run `npx prisma generate`
3. Confirm no errors before continuing

## Build Phase 3 — Vapi Service Module

**File:** `src/services/vapiService.js`

This module contains two things only. Do not add anything else to it.

### 3A. Webhook signature verification

Using the method confirmed in Phase 0:

```js
// verifyVapiWebhook(req)
// - Reads signature from the header confirmed in Phase 0 audit
// - Compares against HMAC of raw request body using VAPI_WEBHOOK_SECRET
// - Returns true if valid, false if invalid
//
// Secret missing behavior:
// - If NODE_ENV !== 'production' and VAPI_WEBHOOK_SECRET is not set:
//     log a warning and return true (dev bypass)
// - If NODE_ENV === 'production' and VAPI_WEBHOOK_SECRET is not set:
//     log an error and return false (reject the request)
```

### 3B. Assistant config builder

```js
// buildAssistantConfig()
```

### 4A. `POST /api/phone/webhook`

This is the single endpoint that receives all Vapi webhook events.

```js
// 1. To preserve raw body for HMAC verification, ensure this route uses
//    express.raw() or equivalent before express.json() parses the body.
//    Confirm the correct approach for your Express version.

// 2. Verify webhook signature using vapiService.verifyVapiWebhook(req)
//    If verification fails: return 401 immediately, do not process further

// 3. Parse event type from payload using field name confirmed in Phase 0

// 4. Route by event type:
//    - assistant-request event (confirmed name): return buildAssistantConfig() response
//    - final call event (confirmed name from Phase 0 — may differ from "call-ended"):
//        call handleCallEnded(payload)
//    - all other event types: return 200 { received: true } — do not error
```

### 4B. `handleCallEnded(payload)` — internal async function in phone.js

```js
// Extract fields from payload using paths confirmed in Phase 0 audit.
// Do not assume field names — use only confirmed paths.
//
// Fields to extract:
// - providerCallId
// - callerPhone
// - startedAt, endedAt
// - transcript
// - summary (use Vapi's native summary if available; otherwise null — do not generate)
// - transferStatus: map Vapi's end/transfer reason to one of:
//     "transferred" | "no_answer" | "voicemail" | "not_attempted"
// - answeredByOwner: true only if transfer completed and was answered
//
// Calculate durationSeconds = (endedAt - startedAt) in seconds, if both exist
//
// Store complete raw payload in rawPayload field
//
// IDEMPOTENCY — use upsert, not create:
//   prisma.callLog.upsert({
//     where: { providerCallId: extractedProviderCallId },
//     update: { /* all extracted fields */ },
//     create: { /* all extracted fields */ }
//   })
// This ensures repeated webhook deliveries update the existing record
// instead of throwing a unique constraint error or creating duplicates.
//
// If providerCallId is null/missing from the payload, fall back to create()
// and log a warning.
//
// If the prisma write fails for any reason:
//   - log the error with full detail
//   - still return 200 to Vapi
//   - do not throw — a DB error must never cause Vapi to retry in a loop
```

### 4C. `GET /api/phone/calls`

Admin endpoint — list call logs.

- Support `?date=YYYY-MM-DD` filter (match on `createdAt` date)
- Support `?transferStatus=` filter
- Default sort: `createdAt DESC`
- Return all CallLog fields except `rawPayload` (too large for list view)
- Return 20 records per page, support `?page=` param

### 4D. `GET /api/phone/calls/:id`

Admin endpoint — single call log detail.

- Return all fields including `rawPayload`
- Return 404 if not found

## Build Phase 5 — Vapi Dashboard Configuration

These steps happen in the Vapi dashboard, not in code. Complete them in order before testing.

1. Log in to Vapi dashboard
2. Go to Integrations → add ElevenLabs API key, confirm voice library syncs
3. Go to Phone Numbers → purchase or import a phone number
4. Set the inbound webhook URL for that number to: `https://[your-replit-public-url]/api/phone/webhook`
5. Copy the webhook secret and add it to `.env` as `VAPI_WEBHOOK_SECRET`
6. Set the assistant trigger mode to dynamic config (backend returns config on `assistant-request`)

**The Replit URL must be the public HTTPS URL, not `localhost`. Vapi cannot reach `localhost`.**

## Failure Handling — If Owner Doesn't Answer

For v1, keep this simple. Include this instruction in the assistant system prompt:

> "If your transferCall attempt fails or is not answered, say: 'I wasn't able to reach anyone right now. Please try calling back or send us a text and we'll get back to you shortly.' Then end the call politely."

Log the call with `transferStatus = "no_answer"`.

Do not build callback capture in v1. That is out of scope for this spec and must not be built now.
