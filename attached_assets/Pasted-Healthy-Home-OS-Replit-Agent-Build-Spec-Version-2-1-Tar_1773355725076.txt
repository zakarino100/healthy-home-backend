Healthy Home OS — Replit Agent Build Spec
Version: 2.1
Target: Replit Agent
Purpose: Extend the existing Healthy Home canvassing app into a lightweight operating system for sales, fulfillment, reviews, content capture, dashboards, and Robin daily analysis.

READ THIS FIRST — Instructions for Replit Agent
You are working on an existing app for a home services company called Healthy Home.

Healthy Home is a local exterior cleaning / softwashing business with these core offers:

House Wash
Driveway Cleaning
Bundle = House Wash + Driveway Cleaning
Your job is to inspect the existing codebase first, then extend it cleanly into a practical business operating backend.

Follow these rules exactly
Inspect the current project architecture first.
Extend what already exists instead of replacing it.
Do not rebuild the app from scratch unless absolutely necessary.
Do not add unrelated features.
Build in sensible phases.
Backend correctness, reporting, and workflow integrity matter more than frontend polish.
When something already exists, reuse and improve it.
If the current project stack differs from the suggested stack below, prefer preserving the current architecture where reasonable and document any tradeoffs.
Do not ask clarifying questions unless there is a true blocker. Default to practical implementation choices.
Test each major phase before moving on.

Stack guidance
If the current app already uses a viable stack, keep it.

If backend pieces are missing and a stack choice is needed, prefer this stack only if it aligns cleanly with the current project:

Frontend: React

Workflow logic
When job is completed:

customer enters review queue
send satisfaction request first
if satisfaction score is 4 or 5, send Google review link
if satisfaction score is 1, 2, or 3, send internal feedback form and flag issue
support batch campaigns for previously serviced customers
Important principle
This is a customer satisfaction triage system, not a manipulative review filter.
The goal is:

invite happy customers to leave a public review
route unhappy customers into internal service recovery

Phase 3 — Review workflow and delivery abstraction
Do not build this as email-only.

The review and satisfaction system must be delivery-channel agnostic.

Design principle
Each outbound workflow message should support:

SMS
email
manual logging
future provider integrations
If no provider is configured yet:

still update workflow state
log the intended outbound payload
mark delivery status appropriately
Required services
Build or extend:

sendSatisfactionRequest
sendGoogleReviewLink
sendFeedbackForm
flagIssue
sendCampaignBatch
Each should:

update workflow state
log outcome
support pluggable delivery adapters
Delivery configuration
Use environment variables like:

REVIEW_DELIVERY_MODE=sms
GOOGLE_REVIEW_LINK=
FEEDBACK_FORM_URL=
Optional provider-specific keys may be added if needed.

Robin delivery design
Do not make email the primary Robin integration.

Instead, build daily report delivery as a modular delivery system with these supported modes:

save to database
expose via API
POST to webhook
optional email fallback
Environment variables
Use:

REPORT_SEND_TIME=20:00
REPORT_DELIVERY_MODE=webhook
ROBIN_REPORT_WEBHOOK_URL=
GOOGLE_REVIEW_LINK=
FEEDBACK_FORM_URL=
Optional:

REPORT_FALLBACK_EMAIL=
RESEND_API_KEY=
Delivery behavior
At end of day:

generate report
save report to database
save canonical JSON payload
if delivery mode includes webhook, POST payload to configured webhook URL
if delivery mode includes email fallback, send readable email summary
store delivery result/status in report record
Important
Design this so that if webhook delivery is unavailable:

report generation still succeeds
failure is logged
delivery status is marked failed
report remains retrievable by API
